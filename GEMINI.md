# MeetLogic MVP 系統規劃書 (Project Blueprint)

## 1. 產品目標與願景
### 1.1 背景與動機
在全球數位轉型中，「會議通膨」已成為損害組織競爭力的頭號殺手。現有工具僅解決「連線」問題，卻造成巨大的管理真空。MeetLogic AI 旨在將會議管理從「軟性勸導」升級為「演算法治理」。

### 1.2 核心目標
*   **數據化治理**：將隱形的溝通損耗轉化為可視化的 ROI 數據。
*   **主動式介入**：賦予 AI 「攔截」與「阻斷」的主動權。
*   **消弭數位債務**：減少無效同步會議，保護員工的深度專注時間。

## 2. 目標用戶與痛點分析
| 用戶角色 | 核心痛點 |
| :--- | :--- |
| **中高階主管** | 缺乏量化手段追蹤高階人才的時數損耗，存在「管理黑洞」。 |
| **一般員工** | 陷入「陪榜文化」與「狀態劇場」，需在深夜償還白天的專注力赤字。 |
| **HR/ESG 部門** | 無法量化員工福祉改善與數位工作產生的碳足跡。 |

## 3. 系統架構配置 (MVP 階段)
MVP 階段將專注於 **Google Meet** 平台，並利用**假資料**快速驗證核心的 AI 攔截與分析價值。

*   **前端 (Client)**：Vue 3 + Vite 開發的 Chrome 擴充功能（直接注入 Google Meet 網頁）。
*   **後端 (Server)**：Python FastAPI。
*   **資料庫 (Database)**：PostgreSQL（MVP 階段存放薪資、職級等假資料）。
*   **AI 引擎**：
    *   OpenAI Whisper (語音轉文字 STT)
    *   GPT-4o (語意分析與決策萃取)
    *   TensorFlow.js (前端本地端臉部/視線追蹤)

## 4. 四階段開發藍圖 (Roadmap)

### Phase 1: 基礎架構建立與資料庫準備 (Scaffolding)
*   **後端**：建立 FastAPI 專案骨架，設定 WebSocket 基礎路由，並配置 PostgreSQL 資料庫連線。
*   **前端**：初始化 Vue 3 專案，並將其配置為 Chrome 擴充功能的格式 (設定 `manifest.json` 等)。

### Phase 2: 事中階段核心 - Google Meet 整合與專注力追蹤
*   **畫面注入**：將 Vue 3 擴充功能的 UI（側邊欄或懸浮窗）成功注入到 Google Meet 頁面中。
*   **邊緣運算 (隱私保護)**：在前端整合輕量級的 TensorFlow.js 模型，直接在瀏覽器端離線分析使用者的臉部微表情和視線，計算出「專注力分數」。
*   **音訊擷取與串流**：透過網頁 API (`getDisplayMedia` 或直接擷取麥克風) 取得會議音訊，並透過 WebSocket 即時串流至 FastAPI 後端。

### Phase 3: AI 分析與 MHI (會議健康指數) 儀表板
*   **語音轉錄**：後端接收音訊串流後，串接 OpenAI Whisper API 將其轉換為逐字稿。
*   **MHI 演算法**：後端綜合前端傳來的「專注力分數」與語音活動狀態，即時計算簡化版的 **MHI (Meeting Health Index)**。
    *   *MHI 公式預設：MHI = w1(專注力) + w2(決策轉化率) + w3(發言平衡度)*
*   **強制冷靜機制**：當後端計算出的 MHI 低於危險閾值時，前端 UI 即時觸發強烈的視覺警告（例如：紅色閃爍畫面或警告標語），展示「主動介入」的概念。

### Phase 4: 事前預算與事後自動化 (MVP 視覺驗證)
*   **事前 (AI 預算守門員)**：在擴充功能的彈出介面 (Popup) 中，使用者可以模擬勾選與會者的職級。系統會從 PostgreSQL 撈取假資料中的對應薪資，即時展示這場會議的「每分鐘燒錢速率」與「預估總成本」。
*   **事後 (零點擊自動化展示)**：會議結束後，後端將完整的逐字稿送交 GPT-4o，萃取出「決策節點」與「行動承諾」。前端 UI 將精美地展示由 AI 生成的「即將送出的 Email 草稿」與「預計在 Jira 建立的任務卡片」，證明系統具備自動化善後的能力。
